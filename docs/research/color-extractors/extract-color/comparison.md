# 2つの実装の比較

anicolors 独自実装と extract-colors ライブラリを対比し、
oshicolor での実装を検討するうえで参考になる観点を整理する。

---

## アーキテクチャ比較

| 観点         | anicolors 独自実装                 | extract-colors ライブラリ                     |
| ------------ | ---------------------------------- | --------------------------------------------- |
| 実装規模     | 約 150 行（単一ファイル）          | 約 400 行（複数モジュール）                   |
| 色空間       | RGB のみ                           | RGB + HSL の2段階                             |
| 量子化方式   | 24刻みの固定グリッド               | 上位4ビット抽出（16刻みバケット）             |
| 近似色マージ | なし（量子化で代替）               | RGB MAE 距離でマージ + HSL 空間でさらにマージ |
| ソート軸     | 頻度（貪欲法で多様性を確保）       | intensity × 面積の少なさ                      |
| 位置情報     | 保持する（画面上のピン座標に使用） | 保持しない                                    |
| 実行環境     | ブラウザのみ                       | Browser / Worker / Node.js                    |

---

## クラスタリング戦略の違い

### anicolors: Farthest Point Sampling

```
候補20色 → 頻度1位を固定 → 残りから「既選択色群との最小距離が最大」の色を選ぶ × N回
```

**長所**: 色相が分散した多様なパレットが得られる。赤・青・緑のように
バランスよく散らばった色を意図的に選べる。

**短所**: 頻度の高い「重要な色」が必ずしも選ばれるとは限らない。
貪欲法なので最適保証はない。

---

### extract-colors: 2段階マージ

```
全ピクセル → RGB 距離でマージ → HSL 距離でさらにマージ → intensity × 面積で最終ソート
```

**長所**: 「知覚的に似た色を統合して、残った特徴的な色を返す」という
より自然な抽出ができる。最終ソートで「鮮やかで面積が小さいアクセントカラー」
を優先するため、単色背景の大きな塗りつぶしより注目色が上位に来やすい。

**短所**: ユーザーが意図して多様性を制御しにくい。パラメータが多く
チューニングが必要。

---

## 明度フィルタリングの比較

|            | anicolors                                      | extract-colors                         |
| ---------- | ---------------------------------------------- | -------------------------------------- |
| 方式       | 固定閾値（輝度 < 30 または > 225）             | colorValidator（ユーザーが自由に定義） |
| デフォルト | 黒・白に近い色を除外                           | アルファ > 250（透明のみ除外）         |
| 意図       | エディタテーマで使いにくい極端色を最初から排除 | 汎用的。テーマ用途なら要カスタマイズ   |

oshicolor でも Neovim カラースキーム用途では明度の極端な色を排除したい。
extract-colors を使う場合は `colorValidator` に明度フィルタを追加するのが適切。

---

## oshicolor への応用指針

### 参考にすべき点

1. **extract-colors の HSL マージ** — 視知覚的に似た色を1つにまとめる処理は
   必須。RGB 距離だけでは「見た目が近い色が重複して出力される」問題が起きる。

2. **anicolors の明度フィルタ** — `brightness < 30 || brightness > 225` の
   フィルタはそのまま採用できる。エディタの背景色や UI 色に使えない極端な色を
   最初から除く設計は理にかなっている。

3. **extract-colors の intensity ソート** — `(intensity + 0.1) × (0.9 - area)` の
   ソート式は、カラースキームで「主役になる特徴的な色」を優先するうえで参考になる。

4. **anicolors の位置情報保持** — oshicolor では画面上にカラーポイントを表示する
   要件があるため、anicolors のように出現座標を記録しておく設計が必要。

### 注意すべき点

- **量子化ステップ数のトレードオフ**: anicolors の 24 刻み（11段階）は粗め。
  細かすぎると似た色が多数残り、粗すぎると異なる色が同一視される。
  16 刻み（extract-colors 相当）が一般的なバランスとされている。

- **サンプリング数**: anicolors の「8px 飛ばし」は固定だが、extract-colors の
  `pixels` オプションは画像サイズに応じて自動調整する。大きな画像では
  後者のアプローチのほうが処理時間が安定する。

- **Hue 距離の円環考慮**: AverageGroup の `hueDistance` 実装（0.5 オフセット法）は
  独自実装時も必ず組み込む必要がある。欠かすと赤系の色が2つのグループに
  分断される問題が起きる。
