# wallust 調査レポート

oshicolor の実装で参考にするため、wallust のコード構造と色抽出の仕組みを調べた。アニメキャラ画像から Neovim テーマを作るときに、どこをそのまま借りられて、どこをアニメ画像向けに作り変えるべきかを整理する。

---

## 1. 全体の流れ

wallust は 1 枚の画像を 16 色のパレットへ落とし込むまでに、画像読み取り、色抽出、16 色割り当て、後処理、という 4 つの段階を順に通す。各段階は Rust の enum で実装が切り替えられるようになっていて、設定ファイルから自由に組み合わせられる。これが「同じ画像から何十種類ものテーマが作れる」という wallust の売りを支えている。

各段の責務は次のようになっている。

| 段階        | 役割                                  | 入力        | 出力                |
|-------------|---------------------------------------|-------------|---------------------|
| Backend     | 画像をどう読んで画素列にするか        | 画像のパス  | RGB の `Vec<u8>`    |
| ColorSpace  | 画素列から代表色を取り出して並べ替える | RGB         | ソート済み色の配列  |
| Palette     | 代表色を 16 個の ANSI スロットへ割り振る | 色の配列  | `Colors` 構造体     |
| 後処理      | コントラストや彩度を整える             | `Colors`    | `Colors`            |

中心となる関数は `lib.rs` の `gen_colors` で、ここが各段階を呼び出してつないでいる。段階ごとにキャッシュ層が挟まっていて、設定を変えていない段は前回の結果を再利用する作りになっている。たとえば Backend だけ変えた場合は色抽出と割り当てだけが再計算される。

主要なソースファイルの役割は次のとおり。

- `src/backends/` は画素取得方式が 6 種類入っている
- `src/colorspaces/` は色抽出のアルゴリズムが Lab、Lch、LchAnsi、LchSalience の 4 種類ある
- `src/palettes/` は Dark、AnsiDark、SoftDark などの 16 色割り当てルールが並んでいる
- `src/colors.rs` には 16 色を保持する `Colors` 型と、彩度補正やコントラスト補正のメソッドが定義されている
- `src/template/` は minijinja と pywal 互換テンプレートの実装である

## 2. Backend が担うこと

ここで言う Backend は、画像ファイルをデコードして RGB バイト列にするまでの処理を指す。色抽出ロジックではなく、その手前の画像入力を切り替えるためにある。なぜこの段が独立しているかというと、画像ファイル全体を読むのか、リサイズしてから読むのか、Kmeans で先にクラスタリングしてから読むのか、で後段の負荷と精度が大きく変わるからだ。ユーザーが速度と品質のトレードオフを選べるようにしてある。

実装されている方式は次のとおり。

| 名前         | やっていること                                            | 速度と精度       |
|--------------|----------------------------------------------------------|------------------|
| `Full`       | デコードして全画素を返す                                  | 遅いが精度は最高 |
| `Resized`    | アスペクト比を保ったまま縮小する                          | 中間             |
| `Wal`        | ImageMagick の `convert` を呼ぶ。pywal 互換のため         | 遅い             |
| `Thumb`      | 512x512 のサムネイルに固定して返す                        | 速い             |
| `FastResize` | SIMD 対応の `fast_image_resize` で縮小する。デフォルト    | 速い             |
| `Kmeans`     | Lab 色空間で K=6 のクラスタリングをかけ、各画素を重心へ写す | 中間             |

oshicolor から見て注目したいのは Kmeans 方式だ。`backends/kmeans.rs` の実装は短く、RGB を Lab に変換してから `kmeans_colors` クレートで重心を取り、画素を重心に写すだけになっている。乱数シードを `0xBEEF` で固定しているので、同じ画像なら必ず同じ結果になる。アニメ画像のように色面がベタで分かれている素材は Kmeans との相性が良いので、oshicolor のデフォルト Backend はこれをベースにすると素直になりそうだ。ただし K=6 はアニメ画像には少なすぎるので、8 から 12 くらいに増やしたほうがキャラの主要色を取りこぼさない。

## 3. ColorSpace の中身

ColorSpace はこのリポジトリで一番濃いところで、画素列から代表色を取り出して順序を決めるまでを担う。RGB のままだと色の近さを人間の感覚に合わせて測れないので、Lab か Lch という知覚色空間に一度変換してから処理するのが共通の方針になっている。

### 共通の処理パターン

`BuildHisto` というトレイトに、すべての色空間が従うべき手順が並んでいる。具体的には次の流れである。

1. `read` で RGB バイト列を Lab か Lch のベクタに変換する
2. `filter_cols` で明度や彩度が極端な色を取り除く
3. `gather_cols` で簡易ヒストグラムを作る
4. `dedup_cols` で似た色をマージしてカウントを足し合わせる
5. `sort_col` で並べ替える
6. `to_rgb` で RGB に戻して返す

このうち `gather_cols` の実装が wallust の色抽出の中心になっている。中身は次のような単純なループだ。

```rust
'outter: for c in colors {
    for hist in &mut histogram {
        if c.col_diff(&hist.color, threshold) {
            if mix { hist.color = hist.color.mix(c, 0.5); }
            hist.count += 1;
            continue 'outter;
        }
    }
    histogram.push(Histo { color: c, count: 1 });
}
```

新しい画素を見たら、すでに記録した色と比較して、しきい値より近ければ既存の色のカウントを足す。遠ければ新しい色として追加する。色の近さは Improved CIEDE2000 という色差式で測っていて、これが知覚的な「色の近さ」を妥当に再現してくれる。

`mix` フラグが立っている色空間では、近い色を見つけたときに既存の色との中間に重心を動かす。`LabMixed` と `LchMixed` がこれを使っており、結果がより滑らかになる。

### Dynamic Threshold

しきい値の選び方も工夫されている。`run_dynamic` という関数がマルチスレッドで複数のしきい値を並列に試して、最終的に色数が 6 から 16 に収まるしきい値を採用する。探索順は 14 を起点にして上下に広げていく擬似的な二分探索で、どの画像でも安定して動くようにしてある。なぜこういう作りになっているかというと、画像によって色の分布が大きく違うからだ。風景写真と単色背景のロゴでは、同じしきい値でも結果がまったく変わってしまう。oshicolor でもアニメ画像は分布の偏りが激しいので、この動的探索の発想は借りる価値が高い。

### 色空間ごとの違い

`Lab` は最もシンプルで、明度 L だけでソートし、`L >= 4.5 && L <= 95.5` の範囲外を捨てる。

`Lch` は wallust のデフォルトで、Lab に加えて彩度フィルタを持つ。`MIN_CHROMA = 10.0` を下回るくすんだ色は捨てて、ソートも明度と彩度のタプルで行う。これによって同じ明度の色が複数あるときは彩度が高い方を優先するため、結果的に鮮やかな色が前に出てくる。

`LchAnsi` はとくに目を引く実装で、`gather_cols` 自体を完全に書き換えている。色相を 6 個のバケットに分けて、画像から各バケットの代表色を取り出す。具体的には次のような割り当てだ。

```text
red     :   0..= 60°
yellow  :  61..=120°
green   : 121..=180°
cyan    : 181..=210°
blue    : 211..=280°
magenta : 281..=360°
black   : L < 5
gray    : L > 95
```

ここで重要なのは、画像にそのバケットの色が存在しなくても、固定値とその近傍から人工的な色を合成して必ず 8 色そろえる点だ。たとえばキャラが青髪と白服だけの画像でも、red と green と yellow のスロットが空にならない。`color_generator` が `unreachable!()` で書かれているのは、ここに到達するはずがないと前提しているからだ。これは ANSI 16 色の順序を必ず守りたい場合の解で、oshicolor のように色相が偏った画像を扱うときにそのまま使える発想になる。

`LchSalience` は人間の注意を引きやすさを擬似的に数値化してソートに使う方式である。明度、彩度、色相のそれぞれにスコアを定義して、重み付き和を取る。スコア式は次のような形をしている。

```rust
salience = w_l * S_l(L) + w_c * S_c(C) + w_h * S_h(h)
S_l(L) = exp(-(L - L_optim)^2 / (2 * sd^2))   // 最適明度に近いほど高い
S_c(C) = C / (C + k)                          // 彩度の有理関数
S_h(h) = 0.5 * (1 + cos(h - p))               // 暖色か寒色かの好み
```

LightFirst では暖色寄りの中明度を、DarkFirst では寒色寄りの低明度を好む重みになっている。oshicolor で「キャラの推し色を優先する」ような目的にはこの形が転用しやすい。重みを差し替えれば「画像中央にある色を優先」「全体平均から離れた珍しい色を優先」といったルールに変えられる。

### 色不足時のフォールバック

抽出後の色が 6 個に満たないときは、`fallback_generator` が補間か補色生成で穴埋めをする。`Interpolate` は 2 色を線形に混ぜる方式で、`Complementary` は HSV 補色を加える。アニメ画像はそもそも色数が少ないことが多いので、この補完は実用上ほぼ必ず走ると考えておいたほうがよい。

## 4. Palette の役割

Palette は抽出した色を `color0` から `color15` までの 16 個のスロットに割り振る。なぜ抽出と分かれているかというと、同じ抽出結果から「明背景のテーマ」と「暗背景のテーマ」と「補色テーマ」が作れて、ユーザーが好みで選べるからだ。つまり ColorSpace と Palette は M 種類 × N 種類の組み合わせとして機能している。

`Colors` という構造体が 16 色とその他のスロットをまとめて持つ。フィールドは pywal 互換で、`cursor`、`background`、`foreground`、`color0` から `color15` までが並ぶ。

代表的な palette を順に見ていく。

`dark` はデフォルト実装で、最も明るい色を `color6` に、最も暗い色を `color1` 側に当てる。背景色は最も暗い色を Lch 空間で 80% 脱彩度してから明度に応じて段階的に暗化する。前景色は最も明るい色を 65% 明るくする。pywal の挙動を踏襲した穏当な選び方になっている。

`ansidark` は `LchAnsi` 色空間と組み合わせて使う専用 palette で、抽出側が ANSI 順に 8 色を返してくれる前提で、`c[0]` を background と `color0` に、`c[1]` から `c[6]` を `color1` から `color6` に、`c[7]` を `color7` と foreground に流し込む。色数が 6 しかないときは `color5` と `color6` を既存色のブレンドで埋める。

`harddark` は dark とほぼ同じだが、背景色を「最頻色を 65% 暗くしたもの」に置き換える。これにより主役色が背景に染み出すような印象になる。

`saliencedark` は salience 順に並んだ色を均等にサンプリングして 6 色にし、それを dark スタイルで割り振る。

### 16 色拡張の工夫

通常の ANSI 16 色では `color9` から `color14` が「明るいバージョン」を担うことになっているが、wallust の `to_16col` メソッドはこれと反対の発想をとる。`color1` から `color6` を 25% 暗くして、`color9` から `color14` のほうを「鮮やかな本来の色」として残す。pywal16 と同じ手法で、画像から取れる鮮やかさを最大限活かすための割り切りである。

### 補色変換

`to_comp` は `color1` から `color6` と `color9` から `color14` を HSV 補色に置き換える。同じ画像から色相を反転させたパレットを作るためのもので、ベースのパレットと組み合わせて多様性を増やすために使われる。

### 後処理

`gen_colors` の最後に呼ばれるのが `postcolor` で、彩度補正と WCAG 21 によるコントラスト補正を行う。`check_contrast_all` は背景と前景や各色のコントラスト比が基準を満たすまで、最大 10 回ほど暗化と明化を繰り返す。これがあるおかげで、抽出結果がどんな極端な色でもターミナルで読める状態に着地してくれる。oshicolor のようなエディタテーマでも、この補正は省けない。

## 5. 抽出アルゴリズムの全体像

ここまでの内容をひとまとめにすると、wallust の色抽出は次のような疑似コードに集約できる。

```
画素 = backend(画像)
lch  = 画素を知覚色空間に変換
lch  = filter_cols(lch)            // 極端な明度や彩度を捨てる
ヒスト = gather_cols(lch, しきい値, mix)  // CIEDE2000 でしきい値ヒスト
ヒスト = dedup_cols(ヒスト, しきい値)      // 類似色をマージ
if ヒスト.len < 6 then ヒスト = fallback(ヒスト, gen)
ソート済 = sort_col(ヒスト, 順序)
カラーズ  = palette(ソート済, 元順序)
postcolor(カラーズ)
```

設計上の工夫は 4 つある。

ひとつめは、各段階の出力を `Vec<u8>` や `Vec<Srgb>` という単純な型にしてキャッシュしやすくしている点だ。これにより設定変更時の再計算量を最小化できる。

ふたつめは、ColorSpace と Palette を独立にして組み合わせの自由度を確保している点である。少ない実装でテーマの種類を爆発的に増やせる。

みっつめは、しきい値選びをマルチスレッドの並列探索に任せて、画像差をアルゴリズム側で吸収していることだ。設計を単純に保ったまま頑健性を稼いでいる。

よっつめは、`LchAnsi` のように `BuildHisto` の標準実装をすべて書き換えて、ANSI 6 色プラス黒灰色の出力を保証する逃げ道を用意していることだ。これがあるおかげで、画像の色相がどう偏っていてもパレットが壊れない。

## 6. oshicolor への展開

oshicolor はアニメキャラ画像から Neovim テーマを生成するという目的があるので、wallust とは前提が違う部分がある。アニメ画像はベタ塗りの単色面が大きく、色相が偏りがちで、背景が単色やグラデで主張が弱い。出力は ANSI 16 色そのままではなく、Neovim のシンタックスグループへのセマンティックな割り当てが必要になる。これらを踏まえて、wallust から借りるべきところと作り変えるべきところを整理する。

### 推奨スタック

| 段階       | 推奨実装                                                       |
|------------|----------------------------------------------------------------|
| Backend    | Kmeans を K=8 から 12 に拡張し、背景マスキングを前段に追加する  |
| ColorSpace | LchAnsi 方式をベースに、Salience スコアを補助的に使う           |
| Palette    | Neovim 用に独自の enum を作り、ターミナル ANSI も同時出力する   |
| 後処理     | WCAG コントラスト補正を必須にする                               |
| Fallback   | Interpolate を採用する。LchAnsi 方式なら欠損 hue を中央値で生成 |

### アニメ画像の特性に合わせた抽出

アニメ画像は色面がはっきり分かれているので、しきい値ヒストよりも Kmeans のほうが直接的に主要色を取れる。さらに画像の四隅の色から flood fill で背景を切り、前景のキャラ部分だけを Kmeans に渡せば、背景色に引きずられずに済む。Kmeans の重心をもう一度 CIEDE2000 で dedup する 2 段構成にすると、ほぼ重複した重心がまとまって安定する。

### 動的しきい値の使いどころ

動的しきい値は oshicolor でも使う前提で考えてよい。役割を明確にしたうえで、Kmeans 側のパラメータと役割分担をさせる形で取り入れる。

なぜ必要かというと、Kmeans の K 値だけでは「似すぎている重心」を統合できないからだ。アニメ画像は髪色のグラデーションや影のように、人間の目には同じ色に見えても Kmeans が別クラスタとして拾ってしまうことがある。このとき K=12 のうち 3 つが「ほぼ同じ髪色」で埋まると、最終パレットの色数が実質的に減ってしまう。CIEDE2000 を使った dedup でこの近すぎる重心を統合する必要があり、その統合の強さを画像ごとに調整するのが動的しきい値の仕事になる。

具体的には次の 2 段構成で使う。

1. Kmeans を K=12 で走らせて 12 個の重心を得る
2. 12 個の重心に対して動的しきい値で CIEDE2000 dedup をかけ、最終的に 6 から 8 色に収める

しきい値の探索範囲は wallust の 2 から 44 ほど広く取る必要はなく、Kmeans の出力に対しては 5 から 20 程度の狭い範囲で十分になる。重心同士の距離はもともと離れているので、しきい値を大きくしすぎると統合が進みすぎてパレットが単調になってしまう。

Kmeans を使わずにしきい値ヒストだけで通す画像も出てくる。たとえばロゴ調のフラットなイラストは Kmeans を経由せずに直接ヒストグラムを作ったほうが軽い。このルートでは wallust と同じく 2 から 30 程度の範囲で動的探索を回し、6 から 12 色に収める。Backend の選択に応じて動的しきい値の探索範囲を切り替える設計にしておくと、画像の種類に応じて挙動が安定する。

### Salience を推し度として再解釈

`LchSalience` の重み式は「キャラの推し色を強調する」用途にそのまま転用できる。たとえば次のような形に拡張できる。

```rust
salience(c) = w_face   * S_face(c)    // 顔領域での出現率
            + w_chroma * S_chroma(c)  // 彩度
            + w_unique * S_unique(c)  // 全体平均からの距離で珍しさを測る
```

顔検出を入れるとコストが上がるので、最初は w_chroma と「画像中心からの距離による減衰」だけで代用しても十分にキャラ色を拾える。

### LchAnsi を中間表現として使う

Neovim テーマでは ANSI 16 色そのままだと粗いので、`LchAnsi` の hue バケット結果を中間表現として使い、そこから Neovim のシンタックスグループへ写す 2 層構造が良い。

```text
画像
  -> LchAnsi 抽出 -> [black, red, green, yellow, blue, magenta, cyan, gray]
  -> Neovim マッピング ->
       Normal       : bg は black 派生、fg は gray 派生
       Comment      : gray を 20% 明るくして低彩度に
       String       : green
       Keyword      : magenta か red。推し色をオプションで優先
       Function     : blue か cyan
       Type         : yellow
       Constant     : red を明るくしたもの
       Visual/Search: 推し色を背景に使い目立たせる
```

この構造の利点は 3 つある。画像に green が無くても LchAnsi が補完してくれるので構文ハイライトが壊れない。推し色フィールドを別に取っておけば Visual や Search のような注目系に自然に当てられる。ターミナル ANSI 16 色も同時に出力できるのでテーマ兼用が成立する。

### コントラスト保証は省けない

エディタテーマで一番大事なのは可読性なので、`Colors::check_contrast_all` の発想はそのまま借りる。

```rust
while !bg.has_min_contrast_text(fg) && i < 10 {
    bg = bg.darken(0.15);
    fg = fg.lighten(0.15);
}
```

加えて、シンタックス色も背景と最低 4.5:1 を確保するループを追加する。アニメ画像から取れる色は淡いものが多いので、ここを省くと Comment が背景に溶けて読めなくなる。

### 出力テンプレート

wallust は minijinja と pywal 互換シンタックスでテンプレートを処理している。oshicolor は Neovim 専用から始めるなら、最初は Lua テンプレートを直書きする方式で十分だ。複数ターミナルへ拡張する段階で minijinja を導入し、kitty、alacritty、wezterm 用のテンプレートも吐けるようにする。Neovim 内で再彩度調整を許したいなら lush.nvim 形式での出力が便利になる。

### 段階的な実装ロードマップ

1. MVP として Kmeans の K=8 と Lab で 8 色を取り、ハードコードされたマッピングで Neovim Lua を出力する
2. LchAnsi 方式に切り替え、hue バケットによる補完を入れる
3. Salience で推し色フィールドを追加し、Visual と Search に適用する
4. Kmeans の重心を CIEDE2000 dedup する 2 段構成にし、動的しきい値で統合の強さを画像ごとに調整する
5. minijinja を導入してターミナル兼用出力に拡張する
6. WCAG コントラスト補正とキャッシュ層、CLI を整える

### wallust から借りないほうがよいもの

`Wal` Backend は ImageMagick 依存なので、Rust や TypeScript で完結させたい oshicolor では採用しない。`Light` palette はアニメ画像で使うと淡くなりすぎて読みにくくなりがちなので、最初は Dark 系だけで十分だ。3 段のキャッシュ層は MVP では不要で、1 枚 1 ファイル出力で始めるのが現実的になる。`/dev/pts` への ANSI シーケンス書き込みは Neovim テーマ生成にはスコープ外なので外す。

---

## 7. 直接読むべきファイル

oshicolor を実装するときに、wallust のコードを直接読んでほしい順に挙げる。

1. `src/colorspaces/lchansi.rs` は hue バケット割り当ての実装で、欠損色を埋めるテクニックがそのまま借りられる
2. `src/colorspaces/lchsalience.rs` は salience スコアの定義と重み調整の実例である
3. `src/colorspaces/mod.rs:445-592` は `BuildHisto` トレイトと `gather_cols` `dedup_cols` の中心ロジックがある
4. `src/backends/kmeans.rs` は Kmeans を palette クレートと組み合わせる最小例として読める
5. `src/colors.rs:367-421` は WCAG コントラスト補正のロジックで、ほぼそのまま流用できる
6. `src/palettes/dark.rs` は背景や前景を画像から導く `getbg` のパターンが書かれている
7. `src/palettes/ansidark.rs` は ANSI 順を保ったまま色をスロットに流し込む例である
8. `src/lib.rs:92-197` はパイプライン全体のキャッシュ込み実装で、設計の流れを掴むのに良い

---

## 8. まとめ

wallust から得られる主な学びは 2 つに集約できる。ひとつは「色抽出は単純なしきい値ヒストグラムと動的しきい値探索だけでも十分に高速で品質が出る」ということで、もうひとつは「`LchAnsi` のように ANSI 順序を強制する設計を入れておくと、画像の色相がどう偏ってもパレットが壊れない」ということだ。

oshicolor では Kmeans と LchAnsi 方式と Salience を組み合わせて使い、Neovim のセマンティックな割り当て層を独自に実装し、WCAG コントラスト補正を必須にする方針にすれば、wallust の頑健性をアニメ画像から Neovim テーマへの変換にうまく持ち込める。
