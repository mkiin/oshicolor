# OKLch 色空間の基礎知識

R2（カラーマッピング）で使う色の原理をまとめたドキュメント。
「なぜ L最小を bg に使うのか」「C とは何か」といった前提知識を補う。

---

## 色空間の選択肢とこのプロジェクトの方針

色を表現する方法は複数ある。

| 色空間    | 値の意味                     | 問題点                                   |
| --------- | ---------------------------- | ---------------------------------------- |
| **RGB**   | R(赤)/G(緑)/B(青) の混合量   | 人間の知覚と一致しない                   |
| **HSL**   | 色相/彩度/明度               | 知覚と一致しない（後述）                 |
| **OKLab** | 明度 L / 赤緑軸 a / 青黄軸 b | 知覚均一だが直感的でない                 |
| **OKLch** | 明度 L / 彩度 C / 色相角 H   | 知覚均一 + 直感的 ← **このプロジェクト** |

### HSL の何が問題か

同じ `saturation=100%` でも、黄色と青では全然違う「鮮やかさ」に見える。
HSL の「彩度」は人間の目が感じる鮮やかさと一致していない。

```
HSL: hue=60, sat=100%, light=50%  → 鮮やかな黄色
HSL: hue=240, sat=100%, light=50% → 鮮やかな青

→ 同じ saturation=100% なのに、青の方が「地味」に見える
```

OKLch ではこの問題が解消されている。同じ C 値なら人間の目には同じ鮮やかさに映る。

## OKLch の3つの軸

```
OKLch(L, C, H)
       │  │  └─ Hue（色相）  0〜360°
       │  └──── Chroma（彩度）0〜約0.4（上限なし、RGBで表現可能な最大が約0.4）
       └──────── Lightness（明度）0〜1
```

### L：明度（Lightness）

- `0` = 完全な黒
- `1` = 完全な白
- 人間の目が「明るさ」として感じる量に対応

```
#1F1F28（暗い背景）→ L ≈ 0.14
#C8C093（薄黄白）  → L ≈ 0.80
#DCD7BA（白に近い）→ L ≈ 0.88
```

**L最小 = 最も暗い色 = エディタ背景（Normal.bg）に最適**
**L最大 = 最も明るい色 = エディタ前景（Normal.fg）に最適**

背景より暗い文字や、文字より明るい背景では可読性が崩壊するため、
この割り当ては「読めるテーマ」の最低条件を満たす。

### C：彩度（Chroma）

- `0` = 無彩色（白/灰/黒）
- 値が大きいほど鮮やか。`0.4` 前後が sRGB の限界
- HSL の saturation と違い、**Hue に依存しない知覚的な鮮やかさ**

```
#727169（灰色がかった色）→ C ≈ 0.02  低彩度
#957FB8（薄紫）          → C ≈ 0.09  中彩度
#7E9CD8（水色）          → C ≈ 0.12  中彩度
#FF5D62（鮮やかな赤橙）  → C ≈ 0.24  高彩度
```

**C最大 = 最も鮮やかな色 = キャラクターの「象徴色」= @keyword に最適**
**C最小 = 最もくすんだ色 = コメント文字（Comment）に最適**

コメントは「背景に溶け込む控えめな色」が慣習。C最小はその役割にぴったり。

### H：色相（Hue）

- `0°` ≈ 赤
- `60°` ≈ 黄
- `120°` ≈ 緑
- `180°` ≈ シアン（水色）
- `240°` ≈ 青
- `300°` ≈ マゼンタ（紫ピンク）
- `360°` = `0°`（循環）

無彩色（C≈0）のときは H が定義されないか無意味な値になる。

## Zone A の割り当てロジックの根拠

```
前処理:
  L最小 → Normal.bg    ← 最暗色。背景に最適
  L最大 → Normal.fg    ← 最明色。前景文字に最適
  残りのC最小 → Comment ← 最くすみ色。控えめなコメントに最適

accents = 残りをC降順で並べる
  accents[0] → @keyword  ← C最大。キャラクター象徴色
  accents[1] → @function ← C2位
  accents[2] → @special  ← C3位
```

### なぜ Hue を参照しないのか（v1→v2 の変更理由）

v1 は「Hue 270〜330° → Keyword グループ」のように Hue レンジで分類していた。
紫/マゼンタ系の単色パレット（全12色が同 Hue 帯）では全色が同一グループになり崩壊した。

C値ランクは Hue に依存しない。**全色が同 Hue 帯でも C の順位は一意**に決まるため、
どんなキャラクターパレットでも壊れない。

## culori での操作方法

```typescript
import { oklch, parse, formatHex } from "culori";

// HEX → OKLch
const color = oklch(parse("#957FB8"));
// → { mode: "oklch", l: 0.53, c: 0.09, h: 305 }

// OKLch → HEX
const hex = formatHex({ mode: "oklch", l: 0.53, c: 0.09, h: 305 });
// → "#957fb8"

// L だけを変えて派生色を作る（shiftL）
const lighter = formatHex({ ...color, l: Math.min(1, color.l + 0.08) });
```

### C の上限と sRGB ガマット

OKLch は sRGB より広い色空間を扱える。C を大きくすると sRGB では表現できない色になる場合がある（ガマット外）。

Zone B で補完色を生成する際は `clampChroma` で sRGB 内に収める。

```typescript
import { clampChroma } from "culori";

// ガマット外の色を sRGB 内に収める（C を自動的に下げる）
const safe = clampChroma(
    { mode: "oklch", l: 0.65, c: 0.3, h: 130 },
    "oklch",
    "rgb",
);
```

## 実際のキャラクターパレット例（紫系）

```
kanagawa テーマ（紫系キャラクターを想定した例）

色           HEX        L     C     H
-------------------------------------------
背景         #1F1F28   0.14  0.01  275°   → Normal.bg  (L最小)
前景         #DCD7BA   0.88  0.03   88°   → Normal.fg  (L最大)
コメント     #727169   0.48  0.02   72°   → Comment    (C最小)
キーワード   #957FB8   0.53  0.09  305°   → @keyword   (C最大, 紫)
関数         #7E9CD8   0.62  0.12  252°   → @function  (C2位, 青)
特殊         #7AA89F   0.67  0.05  185°   → @special   (C3位, 水色)
```

L・C・H を見ると、なぜその色がその役割に割り当たるかが直感的にわかる。

## まとめ：値の大小と意味

| 値  | 小さい                                   | 大きい                   |
| --- | ---------------------------------------- | ------------------------ |
| L   | 暗い（黒に近い）                         | 明るい（白に近い）       |
| C   | くすんでいる（グレーに近い）             | 鮮やか（ビビッドカラー） |
| H   | 色相角（循環）。大小より「距離」で考える |

Zone A の割り当て式を一言で表すと：
**「最暗色を背景に、最明色を前景に、最くすみ色をコメントに、残りを鮮やかさ順にキーワード/関数/特殊に使う」**
