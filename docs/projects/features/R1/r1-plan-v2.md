# 要件更新プラン: R1 色抽出 v2（extract-colors 移行後）

## Context

v1 プランで採用した「貪欲法 + OKLab 知覚距離」実装から、`extract-colors` ライブラリベースの実装に移行した。
また一時期 OKLab / Cinematic の2実装比較構成を取っていたが、両者の差異がほぼなくなったため1本化した。

---

## v1 からの変更経緯

### 採用 → 廃止したライブラリ

| ライブラリ      | 経緯                                                                                                                       |
| --------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `quantize`      | 中央値カット法（MMCQ）実装として採用。`var` スコープバグ・PQueue 不安定ソート・8色超で要求数−1しか返らないバグが判明し廃止 |
| `nearest-color` | Cinematic 実装の色名解決に採用。OKLab 実装に統一したため不要になり廃止                                                     |

### アルゴリズムの変更

| 項目           | v1（貪欲法）                                 | v2（extract-colors）                                |
| -------------- | -------------------------------------------- | --------------------------------------------------- |
| 色抽出         | 独自実装（量子化 → 候補絞り込み → 貪欲選択） | `extract-colors` ライブラリ                         |
| サンプリング   | SAMPLE_STEP=4px 固定ループ                   | `pixels: 40_000`（ライブラリが reducer を自動計算） |
| クラスタリング | RGB 量子化（24 単位丸め）+ グループ集計      | RGB 距離マージ → HSL 距離マージの2段階              |
| 色選択         | 貪欲法（OKLab 距離で多様性最大化）           | `area` 降順スライス                                 |
| 並び順         | 最頻出色 → OKLab 距離最遠順                  | 面積（area）降順                                    |
| 色名解決       | OKLab 距離（culori）                         | OKLab 距離（culori）← 変更なし                      |

## 現在の実装方針

### アルゴリズム: extract-colors + OKLab 色名解決

1. Canvas に画像を原寸描画し `getImageData()` で全ピクセル取得
2. `extractColorsFromImageData(imageData, options)` を呼ぶ
    - `pixels: 40_000` でライブラリ内部のダウンサンプリングを制御
    - `colorValidator` で透明・極暗色・極明色をピクセルレベルで除外
    - `distance: 0.08` / `hueDistance: 0.05` / `saturationDistance: 0.12` / `lightnessDistance: 0.12` で2段階クラスタリングを調整
3. 結果 `FinalColor[]` を `lightness` でフィルタして極端色を除去
4. `area` 降順にソートし上位 12 色を取得
5. `culori` + `color-name-list` で OKLab 距離による色名付与

### 実装定数

| 定数                  | 値     | 理由                                                        |
| --------------------- | ------ | ----------------------------------------------------------- |
| `PIXELS`              | 40,000 | パフォーマンスと精度のバランス点。ライブラリ既定値は 64,000 |
| `DISTANCE`            | 0.08   | デフォルト 0.22 は マージが強すぎて 12 色に届かないため低減 |
| `HUE_DISTANCE`        | 0.05   | デフォルト 0.083 より低く設定し色相の細分化を維持           |
| `SATURATION_DISTANCE` | 0.12   | デフォルト 0.2 より低く設定                                 |
| `LIGHTNESS_DISTANCE`  | 0.12   | デフォルト 0.2 より低く設定                                 |
| `MIN_ALPHA`           | 128    | 半透明以下を除外                                            |
| `MIN_BRIGHTNESS`      | 30     | 極暗色をピクセルレベルで除外（ITU-R BT.601 輝度）           |
| `MAX_BRIGHTNESS`      | 225    | 極明色をピクセルレベルで除外                                |
| `MIN_LIGHTNESS`       | 0.08   | 結果レベルの極暗色フィルタ（HSL 0〜1）                      |
| `MAX_LIGHTNESS`       | 0.92   | 結果レベルの極明色フィルタ（HSL 0〜1）                      |
| 抽出色数              | 12     | 固定                                                        |

### ColorPoint 型定義

```typescript
type ColorPoint = {
    id: number; // 1始まりの連番（area 降順）
    x: number; // 0 固定（extract-colors は位置情報を持たない）
    y: number; // 0 固定
    color: string; // HEX 文字列 "#rrggbb"
    name?: string; // 色名（color-name-list + OKLab nearest 検索）
    percent?: number; // 面積比（area × 100、小数第2位まで）
};
```

## 既知の制約と影響

### 色数を直接指定できない

`extract-colors` は `distance` 等のパラメータで間接的に色数を制御する設計であり、「必ず N 色返す」という保証ができない。

**影響:**

- 色数が少ない画像（背景単色・夜景など）や、フィルタで多く除外される画像では 12 色を下回る場合がある
- Neovim カラースキームは `Normal`・`Comment`・`Keyword` など 12 以上の色ロールへの割り当てを想定している。色が不足すると同じ色を複数ロールに使い回すことになり、テーマのコントラストや識別性が低下する
- 現時点では「不足したまま返す」仕様。将来的には不足時のフォールバック（色の補間・彩度変化で派生色を生成）が必要になる可能性がある

### `x` / `y` 座標が常に 0

`extract-colors` はクラスタの代表色（HEX・面積）は返すが、そのクラスタが画像のどの位置に分布していたかを返さない。

**影響:**

- 「この色は画像上部（髪）から来た色」「この色は中央（顔）の色」といった位置ベースの判断ができない
- 将来的にカラースキームの色ロール自動割り当て（髪色 → アクセント、背景色 → `Normal.bg` など）を実装する場合、位置情報がないと根拠が薄くなる
- 画像上で抽出色の出所を可視化する UI（色スウォッチをクリックすると画像上の対応箇所をハイライト、等）は実装不可
- 現状は `ColorPoint.x` / `ColorPoint.y` 自体は型に残っているが、実質的に意味をなしていない
