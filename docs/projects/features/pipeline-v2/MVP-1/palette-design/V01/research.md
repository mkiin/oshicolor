# MVP-1/palette-design/V01 Research

## パレット生成アルゴリズムの原理

### このドキュメントの目的

`scripts/test-palette-v01.ts` が「なぜそう動くのか」を、色彩理論の基礎から解説する。
コードの各ステップが **どの理論に基づいて** 何をしているかを理解できるようにする。

---

### 1. 使っている色空間: OKLCH と Oklab

#### そもそも色空間とは

色をコンピュータで扱うには、色を数値の組で表現する必要がある。
RGB (#ff0000 = 赤) は最も一般的だが、**人間の知覚と一致しない** という致命的な欠点がある。

```
RGB での「同じ距離」の例:
  #ff0000 (赤) → #00ff00 (緑): R を 255 下げて G を 255 上げた
  #808080 (灰) → #818180 (灰): R を 1 上げて B を 1 下げた

RGB 上の「数値の差」は同じ 255 だが、知覚的な差は全く違う。
前者は激変、後者はほぼ同じに見える。
```

#### OKLCH: 知覚均等な円柱座標

Oklab (Ottosson 2020) は「数値の差 = 見た目の差」になるよう設計された色空間。
OKLCH はその円柱座標版で、3 つの軸を持つ:

```
L (Lightness): 明度   0.0 (黒) 〜 1.0 (白)
C (Chroma):    彩度   0.0 (灰色) 〜 0.4 程度 (鮮やか)
H (Hue):       色相   0° 〜 360° の角度 (赤→黄→緑→青→紫→赤)
```

**なぜ OKLCH を使うか:**

- **L を変えても色味が変わらない** → コントラスト調整で色が壊れない
- **H を変えても明るさが変わらない** → 色相環上の操作が安全
- **C を変えても色相がずれない** → 彩度の上げ下げが安全
- **2 点間の距離 = 知覚的な差** → 「似すぎている色」を数値で判定できる

RGB や HSL ではこれらのどれも保証されない。

#### Oklab: 知覚均等な直交座標

OKLCH の直交座標版。L/a/b の 3 軸で、2 点間のユークリッド距離 (ΔE_ok) が
**知覚的な色の違い** にそのまま対応する。

```
ΔE_ok = √((L₁-L₂)² + (a₁-a₂)² + (b₁-b₂)²)

目安:
  ΔE_ok ≈ 0.02  人間が判別できる最小の差 (JND = Just Noticeable Difference)
  ΔE_ok < 0.08  シンタックスハイライトでは区別しにくい
  ΔE_ok > 0.10  はっきり違う色に見える
```

このスクリプトでは:
- 色相操作 (gap-fill, hue gap) → OKLCH の H を使う
- 距離計算 (弁別性チェック) → Oklab の ΔE_ok を使う
- 明度調整 (コントラスト) → OKLCH の L を使う

### 3. 各ステップの原理

#### §4.0 theme_tone 推定

```
bg の OKLCH L が 0.5 未満 → dark テーマ
bg の OKLCH L が 0.5 以上 → light テーマ
```

dark/light で以降の全パラメータが変わる:
- neutral の L 範囲 (dark: bg 0.10〜0.22 / light: bg 0.92〜0.95)
- accent の L ジッター値
- コントラスト調整の探索方向 (dark: L を上げる / light: L を下げる)
- neutral 派生の L オフセットの符号

#### §4.1 stabilizeHue — なぜ低彩度色の hue を補正するのか

**問題:** OKLCH では C (彩度) が 0 に近い色は hue が不安定になる。
グレーに近い色の hue 値は数学的には存在するが、人間には知覚できない。

```
例: #2d2d31 → OKLCH(L=0.24, C=0.026, H=286°)
    この色は「暗いグレー」であり、紫 (286°) には見えない。
    しかし hue=286° として扱われてしまう。
```

放置すると:
- gap-fill が「286° に色がある」と誤認して、不要な領域を避ける
- primary も 286° の場合、同じ hue に 2 色が集中する

**修正:** 彩度が閾値以下なら、hue を primary の hue に置換し、
C を 0.025 に引き上げて「着色されたグレー (Tinted Gray)」にする。

閾値は L (明度) に応じて変える:
- 暗い色 (L < 0.3): 閾値 0.035 — 暗いと彩度が知覚しにくいため高めに
- 明るい色 (L > 0.85): 閾値 0.025 — 明るくても彩度知覚が低下
- 中間: 閾値 0.015 — Oklab の JND (≈0.02) に基づく

**根拠:** Oklab の JND (Just Noticeable Difference) ≈ 0.02。
C < 0.015 では色相の変化は人間に知覚できない。

#### §4.2 隙間充填 (gap-fill) — 色相環の空き領域を埋める

**問題:** AI が選ぶ 3 色は、キャラの印象に偏る。
赤いキャラなら 3 色とも赤〜橙に集中し、シンタックスハイライトに必要な
多様な色相 (青、緑、紫...) が不足する。

**アルゴリズム:**

```
1. 3 色の hue を 360° の色相環にプロットする

   例: primary=40°, secondary=80°, tertiary=230°

        40°    80°
         ↓      ↓
   0° ───●──────●──────────────────────●─── 360°
                                       ↑
                                     230°

2. 色同士の隙間 (gap) を計算する
   40°→80° = 40°,  80°→230° = 150°,  230°→40° = 170°

3. 最大の隙間の中央に新しい色を配置する
   最大 = 230°→40° (170°) → 中央 = 315° → color4

4. gap を再計算して、次に大きい隙間を分割 → color5
5. 繰り返して color6, color7 を配置
```

**結果:** 色相環上で色が最大限に散らばる。
AI の 3 色がどんな配置でも、4 色追加で 7 色が均等に近い配置になる。

**近接 hue のマージ:** 3 色の hue が 5° 以内に集中している場合
(例: Aglaea 90°, 89°, 90°)、重複したまま gap を計算すると
同じ 360° ギャップが複数カウントされてしまう。
マージして実質 1 色として扱い、残り 360° を均等分割する。

#### §4.3 enforceMinHueGap — 最小色相距離 ≥ 30° の保証

**問題:** gap-fill 後も、AI 3 色と gap-filled 色が 30° 未満に近接する場合がある。
色相が近すぎると知覚的に区別しにくい。

**理論:** Cohen-Or et al. (SIGGRAPH 2006) の色相テンプレート理論。
人間が「調和的」と感じるパレットは、色相環上で一定の間隔パターンを持つ。
最小でも約 30° の間隔があれば、隣り合う色が別の色として認識される。

**アルゴリズム:** バネモデル (force-directed) で実装。

```
各色に「力」を計算:
  近すぎる色から離れる方向に力が発生する
  力の大きさ = (30° - 実際の距離) × 減衰係数 0.3

全ての力を蓄積してから一括更新する (振動防止)
収束するまで繰り返す (最大 20 回)

重要: AI の 3 色 (seed) は固定。gap-filled 色のみ動かす。
```

#### §4.4 L/C 割り当て — Luminance Jittering

**問題:** gap-filled の 4 色に同じ明度 (L) を使うと、
色相は散っていても「全部同じ明るさ」で単調に見える。

**理論:** O'Donovan et al. (SIGGRAPH 2011) — 大規模データセットの分析で、
**調和的なパレットは lightness variance (明度のバラつき) が高い** ことを発見。

**実装:**

```
dark テーマ: L = [0.68, 0.76, 0.72, 0.80]  中心 0.74、ΔL=0.04 ステップ
light テーマ: L = [0.42, 0.50, 0.46, 0.38]  中心 0.44、ΔL=0.04 ステップ
```

**zigzag 割り当て:**
4 色を色相順にソートし、L を「最小, 最大, 2番目, 3番目」の順で割り当てる。

```
色相順: c4=90°  c5=180°  c6=270°  c7=350°
L 割当: 0.68    0.80     0.72     0.76
        最小     最大     2番目    3番目

→ 色相が隣り合う c4(90°) と c5(180°) の ΔL が最大 (0.12)
→ 色相だけでなく明度でも区別しやすい
```

**C (彩度):** AI 3 色の chroma の中央値 × 0.9。
AI の色に馴染みつつ、少し控えめにして gap-filled 色が「脇役」に見えるようにする。

#### §4.5 resolveErrorHue — error 色の hue 衝突回避

error 色は「赤 = 危険」のセマンティクスを持つため、hue を 25° (赤) に固定する。
ただし primary の hue が 25° に近い場合 (例: Hyacine primary=21°)、
accent c1 と error c8 がほぼ同色になってしまう。

```
回避ロジック:
  primary との距離 ≥ 30° → デフォルト 25° を使用
  primary との距離 < 30° → 赤系範囲 (0°〜55°) で最も離れた hue を選択
```

#### §4.6-4.7 neutral 生成 — bg/fg から 10 色を派生

エディタの背景・前景系の色を、AI が提案した bg/fg から L オフセットで派生する。

```
bg 系 (明度の階段):
  bg           = そのまま
  bg_surface   = bg + 0.02  (サイドバー等)
  bg_cursor_line = bg + 0.05  (カーソル行)
  bg_popup     = bg + 0.04  (ポップアップ)
  bg_visual    = bg + 0.08  (選択範囲)

fg 系 (固定 L):
  fg        = そのまま
  comment   = L 0.45 (dark) / 0.55 (light)  — 控えめ
  line_nr   = L 0.40 (dark) / 0.60 (light)  — さらに控えめ
  border    = L 0.30 (dark) / 0.70 (light)  — 目立たない
  delimiter = L 0.60 (dark) / 0.45 (light)  — 少し目立つ
```

light テーマでは L オフセットの符号が反転する (bg が明るいので暗い方向に派生)。

neutral 系は bg の chroma と fg の hue を使い、極めて低彩度にする。
これにより accent 色が視覚的に前面に出る。

#### §4.8 variant — 派生色

```
color1_variant = color1 の C を 0.6 倍 (彩度を落として Tag 用)
color3_variant = color3 の L を ±0.08 (明度違いで Number 用)
```

同じ色相で L/C だけ変えることで「関連する概念」を視覚的にグループ化。

#### §4.9 コントラスト保証 (WCAG AA) + 弁別性自動修正

**コントラスト保証:**
WCAG 2.x の AA 基準 (コントラスト比 ≥ 4.5:1) を満たすよう、
accent 色の L を調整する。

```
dark bg:  L を上方向に探索 (明るくして読みやすくする)
light bg: L を下方向に探索 (暗くして読みやすくする)

step = 0.01 刻みで、最初に 4.5:1 を満たす L を採用。
```

**弁別性自動修正 (fixDiscrimination):**
accent 8 色の全ペア (28 通り) の Oklab 距離を計算し、
ΔE_ok < 0.08 のペアがあれば gap-filled 色の L を ±0.03 調整する。

```
理論: Gramazio et al. (2017) Colorgorical
  シンタックスハイライトのような細い文字では
  ΔE_ok > 0.08〜0.10 が弁別に必要。

アルゴリズム:
  1. 最悪ペア (最小 ΔE) を見つける
  2. 調整可能な側 (gap-filled = c4〜c7) の L を +0.03 と -0.03 の両方試す
  3. ΔE が最も改善する方を採用
  4. 全ペア OK になるまで繰り返す (最大 5 回)
```

#### §4.10 neutral spacing — 同色収束の防止

ensureContrast で L を調整した結果、comment / line_nr / border が
同じ L に収束してしまうことがある (bg が暗すぎ/明るすぎの場合)。

```
最小 ΔL = 0.04 を保証する。
dark テーマ: delimiter > comment > line_nr > border の順で L が下がる
light テーマ: その逆
```

#### §4.11 UI ロール割り当て

AI 3 色のうちどれを「navigation (タブ、フォルダ)」と
「attention (カーソル行番号、Git dirty)」に使うか。

```
navigation = primary をそのまま使う (キャラの最も象徴的な色)
             bg とのコントラスト ≥ 3.0 を保証

attention  = secondary/tertiary のうち、
             primary との Oklab 距離 × chroma が最大のもの
             (primary と最も「違って見える」+「鮮やかな」色)
```

### 4. Gamut mapping — sRGB 範囲への収め方

OKLCH で計算した色が sRGB で表示できるとは限らない (特に高彩度の色)。
sRGB の外にはみ出した色を、知覚的に最も近い sRGB 色に変換する処理。

```
culori.clampChroma() を使用:
  L と H は固定したまま、C (彩度) を二分探索で下げる
  → 色相と明度を保ったまま、彩度だけ落として sRGB 内に収める

これが RGB クランプ (R/G/B を 0-255 にクリップ) より優れている理由:
  RGB クランプ: 色相がずれることがある (#ff4000 → #ff4000 のまま vs #ff0000 に化ける)
  clampChroma:  色相は絶対にずれない (C のみ調整)
```
