# commit

git の変更を Conventional Commits 形式でコミットする。差分の全文を読まず、ファイル別の統計だけで種別を判定する方針を取る。これは過去の実装で `git diff --cached` と `git diff` をそれぞれ 200 行ずつ取得していたためにトークン消費が膨れた経緯への対応として置いている。

## 前処理で取得したコンテキスト

### ステージング状態

!`git status --short 2>/dev/null`

### staged の統計

!`git diff --cached --stat 2>/dev/null`

### unstaged の統計

!`git diff --stat 2>/dev/null`

### 直近のコミットメッセージ

!`git log --oneline -5 2>/dev/null`

## 実行手順

### Step 1: 差分の規模を確認する

ファイル名と変更行数だけで「何が変わったか」をだいたい掴む。差分の中身まで読まなくても、ファイル名の規約で種別を判定できる場合が大半なので、まず stat だけで判断する。

差分が無ければ「コミットする変更がありません」と報告して終了する。

### Step 2: 種別を判定する

ファイル名と stat から種別を判定する。判定の目安は次のとおり。

- ファイルが `docs/` 以下や `*.md` だけで完結している場合は `docs`
- ファイルが `tests/` 以下や `*.test.ts` だけで完結している場合は `test`
- ファイルが設定や依存関係系のみの場合は `chore`
- `src/` 以下に新規ファイルが多数追加されている場合は `feat`
- `src/` 以下の既存ファイル更新が中心の場合は `fix` か `refactor` を会話の文脈から決める

判定がつかない場合のみ、必要なファイルだけを `git diff --cached <file>` で個別に取りに行く。stat だけで決まる場合は full diff を読まない。

### Step 3: ステージングを整える

unstaged の変更がある場合は、意味的なまとまりでステージングする。stat だけで「何がどのコミットに属するか」を判断する。

### Step 4: コミットメッセージを生成する

形式は次のとおり。

```
<種別>: <要約>
```

要約は変更の理由を 1 行で書く。日本語で書いてもよい。複数の論理的変更が混ざっている場合は、Step 3 でコミットを分割してから戻る。

### Step 5: コミットする

```bash
git commit -m "<メッセージ>"
```

## 種別の判断基準

| 種別       | 条件                             |
| ---------- | -------------------------------- |
| `feat`     | 新しい機能・ファイルの追加       |
| `fix`      | バグ修正、既存動作の修正         |
| `refactor` | 動作を変えないコード改善         |
| `docs`     | ドキュメント・コメントのみの変更 |
| `chore`    | 設定・依存関係・ビルドの変更     |
| `test`     | テストの追加・修正               |

## 注意事項

- 全 diff を取得するのはトークンの無駄になる。stat ベースで判断する原則を守る
- `.env` やクレデンシャルを含むファイルが staged されていたら警告する
- 1 コミット = 1 論理的変更を守る。混ざっていたら分割する
- メッセージは簡潔に。本文は原則として書かない
