# docs/projects/features/ 管理規約

## ディレクトリ構造

```
docs/projects/features/
├── R1/                    # フィーチャー単位
│   ├── README.md          # バージョン履歴 + 現行バージョンへのリンク
│   ├── V1/                # バージョン単位
│   │   ├── plan.md
│   │   ├── spec.md
│   │   ├── issues.md
│   │   └── research.md
│   └── V2/
│       ├── plan.md
│       └── spec.md
├── R2/
│   ├── README.md
│   ├── V1/
│   ...
```

## VX/ 配下のファイル命名規則（固定）

| ファイル      | 役割                                                      | いつ書く                           | 必須 |
| ------------- | --------------------------------------------------------- | ---------------------------------- | ---- |
| `plan.md`     | なぜこの版が必要か + 前版の問題 + 設計方針 + 変更対照表   | 実装前（討論の結果をまとめる段階） | Yes  |
| `spec.md`     | アルゴリズム/実装の詳細仕様（コードの「何を・どの順で」） | 実装中〜後                         | Yes  |
| `issues.md`   | この版で判明した課題・限界（次版への引き継ぎ）            | 実装後に問題が見つかったとき       | 任意 |
| `research.md` | 外部調査・業界事例・ライブラリ比較                        | 設計判断の前に調査が必要なとき     | 任意 |

## ルール

- **既存バージョンは削除しない。** 差分確認・設計判断の経緯追跡のため残す
- **既存バージョンへの変更は原則禁止。** 唯一の例外: `issues.md` の追記（次版発行時に `/analyze` が生成する場合を含む）
- **ファイル名にバージョン番号を入れない。** ディレクトリで既に分離されているため冗長
- **VX/ 内のファイルは上記4種のみ。** 別の名前のファイルを作らない
- **README.md は各 RX/ のルートに置く。** バージョン履歴テーブルと現行バージョンへのリンクを持つ

```markdown
| V2  | ...          | ...                       |

## 現行: VX

[`VX/spec.md`](VX/spec.md)

## VX/ 配下のファイル命名規則

| ファイル      | 役割                                   | 必須 |
| ------------- | -------------------------------------- | ---- |
| `plan.md`     | なぜこの版が必要か + 設計方針 + 変更点 | Yes  |
| `spec.md`     | アルゴリズム/実装の詳細仕様            | Yes  |
| `issues.md`   | この版で判明した課題・限界             | 任意 |
| `research.md` | 外部調査・業界事例                     | 任意 |
```

## plan.md のフォーマット

```markdown

前版（VY-1）の問題を簡潔に記述。
issues.md がある場合はそこから引用する。

## 前版との変更対照表

| 項目 | VY-1 | VY  |
| ---- | ---- | --- |
| ...  | ...  | ... |

## 設計方針

設計の核心を記述。

## 変更内容

具体的な変更の詳細。
```

## spec.md のフォーマット

spec.md は自由形式。ただし以下を含めること：

- アルゴリズム/処理の全体フロー
- 主要な定数・型定義
- 依存パッケージ（変更がある場合）

## 開発フロー

```
① スプリント計画: /sprint-plan で今週のゴールと Issue を決める
② 分析（必要時）: /analyze で現状を分析し設計選択肢を討論する
③ 発行（必要時）: /plan で討論結果を plan.md にまとめ、新バージョンを作成する
④ 実装: plan.md または Issue に従って実装する
⑤ 検証: /verification-loop で動作確認。問題があれば issues.md に記録
⑥ 完了: /close-version で spec.md を実装後の状態に更新する
⑦ ふりかえり: /sprint-review で週末にふりかえる
```

### V 発行の基準

- **MVP 完成まで**: V は原則増やさない。横串で繋ぐ作業は Issue で管理する
- **MVP 完成後**: ユーザーフィードバックを元に改善の V を切る
- 設計探索は Issue（`research` ラベル）で追跡する。V ディレクトリは作らない
- V を切るのは **「この設計で実装を完成させる」とスプリントで決めたとき** だけ
