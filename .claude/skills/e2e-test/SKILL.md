# e2e-test

実装完了後の最終検証として、プロジェクト全体が通る状態かを確認する。build、lint、全テストを順に通し、エラーが出たら修正して再実行するループを、すべてグリーンになるまで回す。`/unit-test` が個別関数の TDD サイクルを担うのに対し、こちらは ship 前の全体確認を担う立場にある。

## 入力

引数なし。プロジェクトルートで実行する。

## 実行手順

### Step 1: build を走らせる

```bash
pnpm build
```

成功すれば次に進む。失敗した場合は出力からエラー箇所を特定し、Edit で修正してから再実行する。

### Step 2: lint を走らせる

```bash
pnpm lint
```

成功すれば次に進む。失敗した場合は警告と該当箇所を読む。自動修正で済むなら `pnpm lint --fix` を実行し、それ以外は手動で直してから再実行する。

### Step 3: 全テストを走らせる

```bash
pnpm test
```

成功すれば検証完了。失敗した場合はテストの assertion とエラーメッセージを読み、実装側の問題かテスト側の問題かを切り分けてから修正する。

### Step 4: 動作確認を促す

UI 系の変更が含まれている場合は、`pnpm dev` でローカルサーバーを起動し、ブラウザで挙動を目視確認するようユーザーに案内する。型チェックやテストでは UI の体験そのものは検証できないため、人間の目を最後に通す。

### Step 5: 結果を報告する

ループが完了したら、次の情報を簡潔に伝える。

- 各ステップの最終結果
- 修正したファイル名と要点
- UI 確認が必要かどうか

## 注意事項

- build と lint と全テストは順序を守る。先のステップで失敗したまま次に進まない
- 修正は最小限にとどめる。このスキルの主旨は通すことであり、ついでのリファクタや機能追加は含めない
- 同じエラーで 3 回以上ループしたら自動的に止め、ユーザーに状況を報告して判断を仰ぐ
- 個別関数の TDD サイクルは `/unit-test` で行う。このスキルは ship 前の全体確認を担当する
