# unit-test

新しい関数や機能を実装するときに、テストを先に書いて失敗させ、最小実装で通し、その後にリファクタする TDD サイクルを回す。テストを実装後に書く後付け運用を許さない。後付けは「実装が先に通っている状態」を前提にテストを書くため、保護したい振る舞いではなく現状の実装をなぞるだけのテストになりがちで、回帰検出力が落ちる。

## 入力

`$ARGUMENTS` には実装対象を渡す。たとえば関数名、ファイルパス、機能名のいずれかを書く。

## 実行手順

### Step 1: 仕様を確認する

会話コンテキストから、対象の入力・出力・期待される挙動を 1〜2 文で要約する。曖昧な点があればこの段でユーザーに確認する。実装に入ってから仕様を解釈し直すと、テストが場当たりになり、何を保証しているのかが分からなくなる。

### Step 2: 失敗するテストを書く

実装ファイルに先立って、テストファイルを新規作成または追記する。配置先は `src/**/*.test.ts` か `tests/**/*.test.ts` のどちらかで、対象コードに近い側を選ぶ。テストは仕様の最小ケースを 1 件から 3 件で書き、ハッピーパスとエッジケースを必ず含める。

実装側の関数や型はまだ存在しないので、import 解決時点で失敗する。これは想定内の状態として続行する。

### Step 3: テストを走らせて失敗を確認する

```bash
pnpm test <該当ファイル>
```

期待どおり失敗することを目で確認する。失敗の原因がテスト自体のバグでないこと、つまり import エラーやアサーション不備でないことを確かめる。

ここで予期せず成功してしまった場合、テストが既存実装にぶつかっている可能性があるので、対象を見直してから次に進む。

### Step 4: 最小実装を書く

テストが通る最小限のコードを実装する。設計の汎用化や将来の拡張は意識しない。今書いたテストを通すだけのコードに集中する。汎用化は Step 6 のリファクタで扱う。

### Step 5: テストを走らせて成功を確認する

```bash
pnpm test <該当ファイル>
```

テストが通ったらグリーンの状態で一旦止まる。続けてリファクタに進む。

### Step 6: リファクタする

成功した状態を維持したまま、コードを整える。命名、重複の整理、抽象化のレベル合わせを行う。リファクタ後にもう一度テストを走らせ、グリーンを保っていることを確認する。

### Step 7: ユーザーに報告する

書き込んだあと、ユーザーに次の情報を簡潔に伝える。

- 書いたテストファイルのパスとケース数
- 実装ファイルのパス
- 通したテストの結果

## 注意事項

- テストを後付けで書かない。実装ファイルを開く前に必ずテストを書く
- ハッピーパスだけのテストにしない。少なくとも 1 件はエッジケースを含める
- リファクタ段階で挙動を変えない。リファクタはテストで保護されている範囲のみ
- このスキルは単体テストに限る。プロジェクト全体の最終確認は `/e2e-test` で扱う
