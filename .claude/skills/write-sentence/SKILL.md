# 日本語の文章を書くときのルール

ユーザーに読ませるための日本語の文章を書くときは、英文直訳調を避けて、自然で読みやすい日本語にする。過去にユーザーから受けた具体的な指摘をもとに、表記と構成の両面でルールを定めた。

## 表記のルール

### 文末

文末は「だ・である」調か「です・ます」調のどちらかに統一する。技術ドキュメントの本文は「だ・である」を基本にする。体言止めを使わない。体言止めは英語の名詞句をそのまま日本語にしたような硬さが出るため、文として最後まで書ききる。

```
悪い: バックエンドの選択肢は 6 種類。
良い: バックエンドの選択肢は 6 種類ある。
```

### コロンとカッコ

文末や見出しの末尾にコロン `:` を使わない。「次のとおり」「以下に示す」のような自然な日本語で受ける。

見出しに丸カッコ `()` で補足を入れない。補足したい内容は本文に溶かして書く。本文中の丸カッコも最低限にして、補足が必要なら独立した文として書く。

カギカッコ `「」` の使い分けは適当になりがちなので、強い意図がないなら使わない。引用や強調のために使うときも、同じ文書内でルールを統一する。

```
悪い: ## 抽出アルゴリズム (核心)
良い: ## 抽出アルゴリズムの全体像

悪い: しきい値 (デフォルトは 14) を動的に探索する:
良い: しきい値はデフォルトで 14 から始まり、動的に探索される。
```

### 語彙

カタカナ造語を避ける。「3 段パイプライン」「逆発想」「最終ガード」「逆スキーム」「核心」のような、英語の概念をそのままカタカナや硬い熟語に置き換えた表現は読みにくい。普通の日本語に開く。

普段使わない硬い熟語を使わない。「がいかん」「可読性保障」「直感的に強い」のような表現は技術文書としても不自然なので、もっと素直な言い回しに置き換える。

```
悪い: wallust は 3 段パイプラインで色を抽出する。
良い: wallust は画像から色を取り出すまでに 3 つの段階を順に通す。

悪い: WCAG コントラスト保証の最終ガードがある。
良い: 最後に WCAG のコントラスト基準を満たすよう補正する。

悪い: アニメ画像は Kmeans が直感的に強い。
良い: アニメ画像はベタ塗り面が多いので Kmeans との相性が良い。
```

## 構成のルール

### 章の冒頭で主張を宣言する

各章の最初の 1 文か 2 文で、その章で何を言うのかを宣言してから詳細に入る。いきなり概要や定義に飛び込まない。読み手が「これから何の話をされるのか」を最初に把握できる状態にする。

### 表や擬似コードを出したらその対象を語る

表や擬似コードや図を出したあとは、続く文章でその対象を解説する。表を出してすぐ別の話題に飛ばない。表で大枠を見せた直後に、その表に書いていない別のファイル名や別の概念の説明を始めると、読み手が「この表は何のためにあったのか」を見失う。

### 概念の意味の範囲を最初に定義する

ある言葉の意味する範囲が広いとき、本文で詳細に入る前にスコープを定める。たとえば Backend という言葉が画像入力なのか色抽出なのかが文脈で曖昧なら、「ここで言う Backend は画像をデコードして RGB バイト列にするまでを指す」と先に書いてから個別の実装に進む。

### WHAT より WHY を書く

事実の羅列にしない。それが何かを並べるだけでなく、なぜそういう設計になっているのか、これによって何がうれしいのかを書く。WHY を書かないと読み手にとって「で、それで?」という文章になる。

```
悪い: ColorSpace と Palette は別の enum で実装されている。
良い: ColorSpace と Palette を別の enum にしてあるのは、同じ抽出結果から複数の見た目のテーマを作るためで、これにより少ない実装でテーマの種類を増やせる。
```

### 同じ内容を繰り返さない

表現を少し変えて同じことを言い直すのは冗長になる。要約を兼ねた繰り返しを意図的に置く場合は、章の最後にまとめ節を作って一箇所にまとめる。本文中に何度も同じ説明が出てくる構成は避ける。

### 情報の階層を保つ

見出しレベルと内容の粒度を一致させる。サブ見出しに詳細項目を並べたなら、その詳細はすべて親の見出しの主題に属するものでなければならない。共通の流れを示したあとに、その流れに含まれない別のテーマを同じレベルの小見出しで並べると、読み手が情報の階層を再構築できなくなる。

## 書く前のセルフチェック

文章を書き終えたら、保存する前に次の項目を順に確認する。

1. 文末が体言止めになっていないか
2. 見出しに丸カッコが付いていないか
3. 末尾コロンが残っていないか
4. カタカナ造語や硬い熟語が混じっていないか
5. 各章の冒頭で主張を宣言しているか
6. 表や擬似コードの直後がその対象の解説になっているか
7. 事実の羅列だけで終わっている節がないか
8. 同じ内容が複数箇所に書かれていないか

ひとつでも引っかかったら、保存前に直す。
