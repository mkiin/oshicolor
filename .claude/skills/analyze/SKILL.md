# analyze

フィーチャーの現状を分析し、設計選択肢を提示する。
**このスキルは分析と討論だけで完結する。ファイル作成・Issue・ブランチは別スキルの責任。**

## 前提

docs/ の規約は `docs-convention` スキルに従う。

## 入力

`$ARGUMENTS` は以下の形式:

- `MVP-1/ai-vision` — ステップ名のみ。前版を分析し選択肢を提示する
- `MVP-1/ai-vision "プロンプト設計の改善"` — 概要メモ付き（方針が決まっている場合）

## 実行手順

### Step 1: 前版を分析する

1. `docs/projects/features/pipeline-v2/$0/` 以下の最新バージョン(VXX)を確認する
2. 最新バージョンの `issues.md` があれば読む
3. 最新バージョンの `plan.md` と `spec.md` を読み、現行の設計を把握する
4. `src/features/` 以下の関連コードを Grep/Glob で特定し、主要な実装を確認する

### Step 2: issues.md がなければ生成する

**最新バージョンの `issues.md` が存在しない場合**のみ実行する。

Step 1 で読んだ docs と実装コードから、現バージョンの課題・限界を `issues.md` としてドラフト生成する。

```markdown
# MVP-X/<ステップ名>/VXX Issues

## 判明した課題

| #   | 問題 | 原因・根拠                                            | 重要度          |
| --- | ---- | ----------------------------------------------------- | --------------- |
| 1   | ...  | plan.md の未決定事項 / コード上の制約 / spec との乖離 | High/Medium/Low |

## 限界・トレードオフ

- ...
```

生成後、ユーザーに提示して確認する。
ユーザーの修正を反映した上で保存する。

**`issues.md` が既に存在する場合**: そのまま読んで Step 3 に進む。

### Step 3: 設計選択肢を提示する

**概要メモが引数で渡されている場合**:
概要メモを軸に整理して提示する。

**概要メモがない場合**:
分析結果をもとに以下を提示し、ユーザーと討論する:

```markdown
## 前版(VXX)の問題分析

| #   | 問題 | コード上の原因            | 重要度          |
| --- | ---- | ------------------------- | --------------- |
| 1   | ...  | `src/features/.../xxx.ts` | High/Medium/Low |

## 設計選択肢

### 案 A: [タイトル]

- 概要: ...
- Pros: ...
- Cons: ...
- 影響ファイル: ...

### 案 B: [タイトル]

- 概要: ...
- Pros: ...
- Cons: ...
- 影響ファイル: ...

## 推奨案

[選択理由]
```

### Step 4: ユーザーの判断を待つ

**ここで止まる。** 方針が決まったら `/plan` で docs を作成する。

## 注意事項

- このスキルは **分析と討論のみ**。V ディレクトリの作成、Issue 作成、ブランチ作成は行わない
- 前バージョンのファイルは issues.md 以外変更しない
- 深い分析が必要な場合はユーザーに planner subagent の使用を提案する
