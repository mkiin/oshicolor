# library-research

外部ライブラリの設計思想、API、アルゴリズムなどを調査し、`docs/references/<lib>/overview.md` に構造化されたレポートを残す。oshicolor への展開方針、つまり採用するかしないか、するならどう変えて取り入れるかを必ず含めることで、後でその情報を使って実装判断ができる状態にする。

## 入力

`$ARGUMENTS` の形式は次のとおり。

- `<library-name>` のみのとき: 全般的な調査
- `<library-name> <調査対象>` のとき: 調査対象を絞って深く読む。例として `wallust アーキ`、`jotai async` のような書き方をする

## 実行手順

### Step 1: 調査対象の範囲を確認する

会話コンテキストから「何を知りたいのか」を明らかにする。漠然とした「<ライブラリ> を調べる」だと出力が冗長になり後で読み返さない結果になるので、最初に範囲を 1〜2 文で言語化する。曖昧な場合はユーザーに確認する。

### Step 2: 既存資料を確認する

`docs/references/<lib>/` の存在を Glob で確認する。

- 存在する場合: 既存ファイルを Read し、追記すべきか別ファイルに分けるかを判断する
- 存在しない場合: 新規作成モードへ進む

### Step 3: 情報を集める

調査対象と範囲に応じて、次のソースを使い分ける。

- 公式ドキュメントは WebFetch で読む
- GitHub のリポジトリ構造は `gh repo view` や `gh api` で見る
- 主要ファイルのソースコードは raw URL を WebFetch で取りに行く
- 補完情報は WebSearch で集める

集める内容は次の観点に絞る。これらが揃えば oshicolor への展開判断ができる。

- このライブラリが解く課題と中心となる概念
- 主要なモジュール、型、アルゴリズム
- API の使い方の最小例
- 性能特性と既知の制約
- ライセンスと依存関係

### Step 4: テンプレを読む

`.claude/templates/library-research.md` を Read で読み込む。

### Step 5: 内容を生成する

テンプレに沿って次の構成でまとめる。

1. 全体像。何をするライブラリか、設計の中心となる概念は何か
2. 主要トピックの解説。3〜6 セクションで深掘りする
3. oshicolor への展開。採用方針、推奨スタック、借りないほうがよいもの
4. 直接読むべきファイル。ソースを読んだ場合のみ
5. まとめ

執筆は `/write-sentence` のルールに従う。具体的には、章の冒頭で主張を宣言し、表や擬似コードの直後にその対象を解説し、WHY を必ず書く。

### Step 6: ユーザーに確認する

draft の全文と保存先パスを提示する。修正がなければ Write する。

### Step 7: 結果を報告する

書き込んだあと、ユーザーに次の情報を簡潔に伝える。

- 保存先パス
- 調査の要約として 3〜5 行
- oshicolor への展開で読み返す価値のある節

## 注意事項

- 「使い方の網羅的なリファレンス」は書かない。oshicolor の実装判断に効く情報に絞る。網羅的にしたくなったら公式ドキュメントへのリンクで済ませる
- ソースコード未読のままで「アーキテクチャ」を書かない。読まずに書くと表面的な要約になり、後で実装する側が読み返しても得るものがない
- ライセンスが商用利用と相性が悪い場合は、oshicolor への展開節で必ず警告を出す
- 書き込み先は `docs/references/<lib>/` 配下に限る。features の spec.md には別途 `/spec-write` で反映する
