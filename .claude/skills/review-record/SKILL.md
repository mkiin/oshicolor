# review-record

このスキルは、AI が自分で feature をレビューして指摘を review.md に残すためのものだ。ユーザーが外部のレビュー結果を貼り付ける運用ではなく、AI が spec と実装コードを読み、独立した視点で評価する。レビューの対象はアーキテクチャ・アルゴリズム・システムデザインに限定し、動作不具合や個別バグは扱わない。

## 入力

`$ARGUMENTS` には feature 名を渡す。たとえば `color-extract`、`palette-design`、`lua-gen` のように書く。

## 実行手順

### Step 1: レビュー対象を読み込む

レビューには判断基準と評価対象の両方が必要になる。基準は spec に、対象は実装コードにあるので、両者をまとめて読む。

1. `docs/features/$ARGUMENTS/spec.md` を Read で読む
2. `src/features/$ARGUMENTS/**/*.{ts,tsx}` を Glob で一覧取得し、主要ファイルを Read で読む
3. spec.md が存在しない場合はレビューを中断し、その旨をユーザーに報告する

### Step 2: レビューを実施する

レビューは次の 3 領域に絞る。これ以外の指摘、たとえば動作不具合や個別バグは見つけても review.md には載せず、別途 issue として切り出してもらう方針にする。

- arch は、モジュール分割、責務分離、依存方向、公開 API の境界を見る
- algo は、計算量、エッジケース、数値安定性、ロジックの正しさを見る
- design は、データフロー、状態管理、エラーハンドリング、テスタビリティを見る

spec と実装の乖離、設計上の改善余地、将来の拡張で困りそうな箇所を 3〜5 件挙げる。客観的な観察として書き、重大度の判定や優先度付けはしない。採否はユーザーが後で判断する。

### Step 3: review.md を準備する

ファイルが既存か新規かで処理を分ける。

- 存在しない場合は `.claude/templates/review.md` を Read して feature 名を埋め、新規作成する
- 存在する場合はそのまま続行する

### Step 4: エントリを生成する

各指摘について次のブロックを 1 つずつ作る。複数指摘を 1 ブロックにまとめない。1 ブロック対 1 指摘の対応が崩れると、後で issue に切り出すときに参照点が失われる。

```
### YYYY-MM-DD <自分のモデル名> (review-record)

- **領域**: arch / algo / design
- **指摘**: <要点を 1〜2 文で>
- **対応**: 保留
- **メモ**: <補足や根拠>
```

`<自分のモデル名>` には実行中の自分のモデル名を入れる。たとえば `Opus 4.7` や `Sonnet 4.6` のように書く。判断がつかない場合は `Claude` とだけ書く。
`対応` の初期値は `保留` で固定する。採否はユーザーが判断するため、AI が先回りして決めるとレビューログにバイアスが残る。

### Step 5: prepend して書き込む

既存のエントリは上書きせず、新しい指摘を上に積み上げる方式で書き込む。これにより最新のレビューが見出しの直下に来て、時系列で逆向きに読める並びになる。

### Step 6: 結果を報告する

書き込んだあと、ユーザーに次の情報を簡潔に伝える。指摘の要約を出さないと、ユーザーは review.md を開きに行かないと中身を把握できない。

- 指摘を何件記録したか
- 保存先パス
- 指摘の要約として、各 1 行で 3〜5 件
- issue 化したい指摘があればその対応案

## 注意事項

- 動作不具合や具体的なバグは扱わない。見つけた場合は `/create-issue` でローカル issue を作るようユーザーに案内する。レビューログに混ざると、後で領域ラベルでフィルタしたときに精度が落ちる
- spec が無い feature ではレビューを実施しない。基準が無い状態で書いた指摘は主観的になり、レビューログとして残しても判断材料として使えない
- 領域ラベルとして arch、algo、design のいずれかを必ず選ぶ。3 領域のどれにも当てはまらない指摘は、このスキルの範囲外なのでレビューに含めない
- 1 指摘 = 1 ブロックを守る。同じ領域に複数の指摘があってもブロックを分ける
